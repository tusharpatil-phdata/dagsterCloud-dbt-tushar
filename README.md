4. Pipeline Flow

┌──────────────────────────┐
│  daily_ingestion_schedule │  Cron: 6 AM UTC (or manual trigger)
└────────────┬─────────────┘
             ▼
┌──────────────────────────┐
│  run_ingestion_job        │  Asset: ingest_daily_data
│                           │  Step 1: COPY from ADLS → *_STAGE
│                           │  Step 2: Validate (non-empty, no NULLs)
│                           │  Step 3: Threshold check (anomaly detection)
│                           │  Step 4: MERGE + DELETE sync → SOURCE
│                           │  Step 5: DML audit → METRICS.SOURCE_DML_AUDIT
│                           │  Step 6: Job run + SOURCE counts → METRICS
│                           │  On failure: log FAILURE → METRICS.DAGSTER_JOB_RUNS
└────────────┬─────────────┘
             ▼  sensor: trigger_dbt_after_ingestion (lightweight RunRequest)
┌──────────────────────────┐
│  run_dbt_cloud_job        │  Triggers dbt Cloud: dagsterCloud-dbt-prod-job
│                           │  dbt builds: LZ (bronze) → STAGING (silver) → DBO (gold)
│                           │  On failure: sensor triggers dagsterCloud-dbt-retry-job
└────────────┬─────────────┘
             ▼  sensor: trigger_metrics_after_dbt (lightweight RunRequest)
┌──────────────────────────┐
│  run_post_dbt_metrics_job │  Asset: log_dbt_results_to_metrics
│                           │  Step 1: Log dbt job run → DAGSTER_JOB_RUNS
│                           │  Step 2: LZ/STAGING/DBO row counts → LAYER_ROW_COUNTS
│                           │  Step 3: Per-model dbt results → DBT_MODEL_RUNS
└──────────────────────────┘
             ▼
┌──────────────────────────┐
│  Dagster+ Email Alerts    │  Built-in — success/failure notifications
└──────────────────────────┘


Key Design Decisions







Decision



Why





Sensors are lightweight (no Snowflake calls)



Dagster+ Serverless has 60-second sensor timeout. All Snowflake work happens inside assets.





Audit logging in assets (not sensors)



Assets have no timeout — can do multiple Snowflake queries safely.





SOURCE counts logged by ingestion



Avoids duplicate SOURCE rows in LAYER_ROW_COUNTS.





LZ/STAGING/DBO counts logged by post-dbt asset



Only runs after dbt succeeds — captures the correct post-transformation counts.





Failure logging in both assets and sensors



Ingestion failures logged by the asset's except block. dbt failures logged by the failure sensor.





Single database



METRICS schema inside main DB — no sandbox needed for a POC.

6.3 Generate RSA Key Pair (No Passphrase)

Run these 3 commands separately (in Git Bash or PowerShell with OpenSSL):

# Step A: Generate raw private key
openssl genrsa 2048 > rsa_key_raw.pem

# Step B: Convert to PKCS8 format (what Snowflake/dbt need)
openssl pkcs8 -topk8 -inform PEM -outform PEM -nocrypt -in rsa_key_raw.pem -out rsa_key.p8

# Step C: Extract public key
openssl rsa -in rsa_key.p8 -pubout -out rsa_key.pub


Verify:

head -1 rsa_key.p8    # Should show: -----BEGIN PRIVATE KEY-----
head -1 rsa_key.pub   # Should show: -----BEGIN PUBLIC KEY-----


Files created:





rsa_key.p8 → private key → goes into dbt Cloud + Dagster+ env var



rsa_key.pub → public key → goes into Snowflake ALTER USER



rsa_key_raw.pem → intermediate (can delete)



⚠️ NEVER commit these files to Git. They are in .gitignore.

6.4 Attach Public Key to Snowflake User





Open rsa_key.pub → copy body only (remove BEGIN/END lines, remove all newlines → single string)



Run in Snowsight:

USE ROLE ACCOUNTADMIN;
ALTER USER DAGSTER_DBT_USER_CLOUD
  SET RSA_PUBLIC_KEY = '<single-line-public-key-body>';

-- Verify (RSA_PUBLIC_KEY_FP should NOT be empty)
DESCRIBE USER DAGSTER_DBT_USER_CLOUD;

16. Extending the Pipeline (Add New CSV)

To add a new CSV (e.g. payments.csv), follow these steps:

Step 1: ADLS

Upload payments.csv to raw_data/ in your ADLS container.

Step 2: Snowflake (SOURCE schema)

USE SCHEMA SOURCE;

-- SOURCE table
CREATE OR REPLACE TABLE PAYMENT (ID STRING, ORDER_ID STRING, AMOUNT STRING, METHOD STRING);

-- STAGE table
CREATE OR REPLACE TABLE PAYMENT_STAGE LIKE PAYMENT;

-- Stream
CREATE OR REPLACE STREAM PAYMENT_CHANGES ON TABLE PAYMENT;

-- Threshold config
INSERT INTO METRICS.THRESHOLD_CONFIG VALUES ('PAYMENT', 60, 30, 10);


Step 3: Dagster config.py

Add one TableSpec to the TABLES list:

TableSpec(
    name="PAYMENT",
    csv_file="payments.csv",
    pk="ID",
    key_cols=["ID", "ORDER_ID"],
    merge_set=["ORDER_ID", "AMOUNT", "METHOD"],
    all_cols=["ID", "ORDER_ID", "AMOUNT", "METHOD"],
),


Add to DBT_LAYERS:

("LZ", "RAW_PAYMENT"),
("STAGING", "STG_PAYMENT"),
("DBO", "DIM_PAYMENT"),  # or FCT_PAYMENT


Step 4: dbt Models

Create 3 files:





models/lz/raw_payment.sql



models/staging/stg_payment.sql



models/dbo/dims/dim_payment.sql (or facts/fct_payment.sql)

Add entries to:





_lz_sources.yml (source + model)



_stg_schema.yml (model)



_dims_schema.yml or _facts_schema.yml (model)

Step 5: Deploy

git add -A
git commit -m "feat: add PAYMENT entity end-to-end"
git push origin main


Dagster+ auto-deploys. Run run_ingestion_job → dbt picks up new models automatically.



17. Troubleshooting







Problem



Fix





"Missing dagster_cloud.yaml"



Must be at repo root, pushed before connecting Dagster+





"Multiple top-level packages"



Add [tool.setuptools.packages.find] to pyproject.toml with include = ["dagster_code*"]





dbt project name error



name in dbt_project.yml must use underscores only (no hyphens)





No dbt assets in Dagster



Run the dbt Cloud prod job manually once first





Sensor timeout (60s)



Never make Snowflake calls inside sensors — do it in assets





Key pair auth error



Ensure PEM includes BEGIN/END headers, real newlines (not \n literal)





DAGSTER_DBT_KIEWIT_DB not found



Search-replace all dbt files: change to DAGSTER_DBT_KIEWIT_DB_CLOUD





dbt Cloud IDE shows "Initialize"



Set project subdirectory to demo_kiewit_dbt in dbt Cloud settings





Git push rejected



Run git pull origin main --rebase first, then push





Duplicate SOURCE in LAYER_ROW_COUNTS



Post-dbt asset should use DBT_LAYERS (excludes SOURCE), not ALL_LAYERS





Alerts not arriving



Check Dagster+ → Settings → Alerts → verify email and policy are active





FCT_ORDER_DETAIL empty



Drop the table in Snowflake, re-run dbt (incremental on empty table issue)
